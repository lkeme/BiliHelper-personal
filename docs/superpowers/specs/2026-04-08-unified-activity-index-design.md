# Unified Activity Index Design

**Goal:** Build a Python-based unified activity index pipeline that discovers and structures multiple activity types from a single Bilibili space feed, then feeds `ActivityLottery`, `Lottery`, and `LiveReservation` through dedicated remote resource files.

**Scope:** Add a new Python collector/parser pipeline, generate four `resources/*.json` files, wire the three target plugins to dedicated remote files, and publish updates from GitHub Actions on a fixed schedule.

## Background

The project already has:

- a local helper script at [sync-activity-links-from-space.php](/D:/WorkSpace/PhpCode/BiliHelper-personal-ws/tools/local/sync-activity-links-from-space.php)
- a script plugin `ActivityInfoUpdate`
- an existing remote/local consumption pattern for `ActivityLottery` through `activity_infos.json`

But the current setup is limited:

- it is PHP-only and login-dependent
- it only serves `ActivityLottery`
- it mainly extracts links, not a unified structured activity catalog
- it is not designed for GitHub Actions scheduled updates

The new requirement is broader:

- discover multiple activity types from one space feed
- publish structured remote resources
- let three plugins consume those resources directly
- keep user-side plugin execution steps intact, rather than fully precomputing execution payloads remotely

## Source and Classification

### Space source

Primary source space:

- `https://space.bilibili.com/3461568884902878/dynamic`

### Activity title prefixes

The collector should classify articles primarily by title prefix:

- `活动抽奖圆盘抽奖****`
- `互动抽奖****`
- `预约抽奖****`
- `充电抽奖****`

### Classification strategy

- title prefix is the primary classifier
- article body/content is the validation layer
- if title matches but body structure does not support the expected activity type, the article is skipped

## Technical Stack

The new pipeline is implemented in **Python**, not PHP.

### Reason

- the work is collector/parser heavy
- Python is better suited for this style of structured scraping and transformation
- the existing PHP `ActivityInfoUpdate` should remain as a local helper tool, not become the main automation pipeline

### Existing PHP tools

`ActivityInfoUpdate` remains in the repository, but it is **not** the main path for this new remote-index system.

## Authentication Policy

### Default mode

The main pipeline must run in **unauthenticated mode** by default.

Reason:

- GitHub Actions should not use live account cookies unless absolutely necessary
- this reduces account risk and abnormal-IP exposure

### Optional enhancement

A single optional cookie injection point may be reserved, for example:

- `BILI_COOKIE`

This is only a fallback/enhancement path if a specific activity type later proves impossible to parse well enough without login.

Current design requirement:

- do not depend on cookie for the main path

## Collection Window

Recent article collection window:

- only collect target articles published within the last **24 hours**

Reason:

- the target account typically posts around `00:00-01:00`
- two daily action runs are sufficient
- older articles are not useful as a standing index source

## Collection Flow

The collector follows a three-stage flow:

1. **Dynamic discovery**
- find recent `read/cv...` article links from the target space feed

2. **Article fetch and validation**
- fetch article body/content
- classify by title prefix
- validate against body/content cues

3. **Type-specific parsing**
- parse each article into one of four output datasets

This is a strict pipeline:

- dynamic -> article -> classified structured output

## Output Files

The pipeline generates four separate files in `resources/`:

- `activity_lottery_infos.json`
- `interactive_lottery_infos.json`
- `reservation_lottery_infos.json`
- `charge_lottery_infos.json`

### Why split files

- each plugin should fetch only the data it needs
- avoids cross-plugin noise
- aligns with independent plugin responsibilities

## Shared JSON Envelope

All four files use the same outer envelope:

```json
{
  "code": 200,
  "remarks": "generated by tools/activity_index",
  "data": []
}
```

No historical archive field is required.

## File Schemas

### 1. `activity_lottery_infos.json`

This file replaces the current `activity_infos.json`.

Compatibility priority is high: the existing `ActivityLottery` plugin already works well with the current structure, so the new file should stay close to that model.

Minimum fields:

- `title`
- `url`
- `page_id`
- `activity_id`
- `lottery_id`
- `start_time`
- `end_time`
- `source_cv_id`
- `source_url`
- `publish_time`
- `update_time`

### 2. `interactive_lottery_infos.json`

This file is a structured **hint source**, not a final execution payload.

Minimum fields:

- `title`
- `source_cv_id`
- `source_url`
- `publish_time`
- `draw_time`
- `dynamic_id`
- `dynamic_url`
- `reserve_rid`
- `up_mid`
- `requires_follow`
- `requires_repost`
- `requires_comment`
- `prize_summary`
- `update_time`

### 3. `reservation_lottery_infos.json`

This file is also a structured **hint source**.

Minimum fields:

- `title`
- `source_cv_id`
- `source_url`
- `publish_time`
- `draw_time`
- `up_mid`
- `up_name`
- `sid`
- `name`
- `jump_url`
- `text`
- `requires_follow`
- `requires_reserve`
- `update_time`

### 4. `charge_lottery_infos.json`

No plugin consumes this file yet, but it should still be emitted at the same quality level as the other index files.

It should contain:

- activity identity
- source identity
- time information
- condition information where parseable

## Explicitly Excluded Schema Item

Do **not** include a `status` field.

Reason:

- the files are refreshed daily on a fixed schedule
- plugin-side runtime logic can decide freshness and validity using current time plus activity timestamps

## Plugin Integration

### `ActivityLottery`

- rename the local/remote source file from `activity_infos.json` to `activity_lottery_infos.json`
- do not preserve old filename compatibility
- keep the effective data structure close to the current one

### `Lottery`

`Lottery` fully switches to the new remote index file:

- `interactive_lottery_infos.json`

But this switch does **not** mean the remote file becomes an execution payload.

The user-side plugin must still perform its own critical steps, such as:

- dynamic detail confirmation
- requirement checks
- final reservation execution flow

The remote file is a structured discovery source, not a full action replay source.

Primary external reference for this plugin-side boundary:

- `reference/bilibili-API-collect-trunk/docs/dynamic/detail.md`

### `LiveReservation`

`LiveReservation` consumes:

- remote `reservation_lottery_infos.json`
- local configured `vmids`

The local `vmids` path remains supported.

Deduplication rule:

- dedupe by **uid / up_mid**

As with `Lottery`, the plugin must still keep its own critical execution-side validation and API interaction.

### `charge_lottery_infos.json`

- generated now
- no runtime plugin consumption yet

## Remote Resource Strategy

The generated files are committed directly into the current repository branch used for remote resources.

No separate history archive is needed.

If old versions are needed later:

- use git history

## GitHub Actions Strategy

### Location

GitHub Actions run in the **current repository**, not a separate private repo.

Reason:

- default flow is unauthenticated
- no account cookie is required in the normal path
- keeping workflow, code, and generated resources together simplifies maintenance

### Triggers

- cron at `00:40` Asia/Shanghai
- cron at `12:40` Asia/Shanghai
- `workflow_dispatch`

### Update policy

- run collector/parser
- update the four `resources/*.json` files
- commit only if contents changed

### Commit strategy

- one combined commit for all changed index files
- fixed commit message:
  - `chore: refresh remote activity indexes`

## Behavioral Boundaries

The remote index system must not overreach into user-side execution.

### Allowed remotely

- article discovery
- activity classification
- extracting structural hints
- extracting basic time windows
- extracting basic condition hints

### Not allowed remotely

- turning every activity into a final user-execution payload
- skipping plugin-side critical API confirmation
- replacing user-side execution flow entirely

This is especially important for:

- `Lottery`
- `LiveReservation`

Those plugins must still behave like plugins operating with the current user environment, not like dumb replay clients of a remote script.

## Non-Goals

Out of scope for this phase:

- a `ChargeLottery` runtime plugin
- historical JSON archives
- preserving `activity_infos.json` legacy filename compatibility
- making remote files contain final action payloads for all plugins
- making login-cookie scraping the default workflow
